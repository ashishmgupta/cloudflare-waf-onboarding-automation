Cloudflare Zone Onboarding Automation

Production-grade automation for Cloudflare Enterprise zone creation and WAF/logging configuration via the Cloudflare REST API v4.

1. Overview

This script (cloudflare_zone_setup.py) fully automates the creation of a Cloudflare zone and the application of a standardised Enterprise security and logging configuration across 12 sequential steps. Every API call is verified, and any failure triggers automatic rollback with a full HTML diagnostic report.

What it does

  • Creates a zone under your Cloudflare Enterprise account
  • Adds a proxied A or CNAME DNS record (required for WAF and all security features)
  • Applies security settings: TLS 1.2 minimum, HTTP/2 off, 120s origin timeout
  • Enables log retention (72-hour Logpull window)
  • Deploys the Cloudflare Managed WAF Ruleset in log mode
  • Disables caching for all requests
  • Creates 4 Logpush jobs: HTTP requests, firewall events, and DNS logs to Splunk; HTTP requests to Dynatrace
  • Generates a full HTML report on completion (success or failure)

Architecture

cloudflare_zone_setup.py │ ├─── reads ──────────────────► credentials.txt │ │ POST /zones ├──────────────────────────► Cloudflare API v4 │ │ │ Zone created │ │◄───────────────────────────────────┤ │ │ │ (12 sequential steps) │ ├──────────────────────────────────► │ │ │ │ ┌─── Zone Settings │ │ │ TLS 1.2 min │ │ │ HTTP/2 off │ │ │ Origin timeout 120s │ │ │ Log retention on │ │ │ │ │ ├─── WAF │ │ │ Managed Rules (log mode) │ │ │ │ │ ├─── Cache Rules │ │ │ Disable all caching │ │ │ │ │ └─── Logpush Jobs (4 max) │ │ HTTP → Splunk │ │ Firewall → Splunk │ │ DNS → Splunk │ │ HTTP → Dynatrace │ │ │ └─── generates ──────────────────► zone_report_{zone}_{ts}.html
⚠ Zone activation requires manual action Creating a zone via the API does not activate it. You must update your domain registrar's nameservers to point to the Cloudflare nameservers assigned to the zone. Until this is done, DNS and all security features will not be active.

2. Prerequisites

Account requirements

  • Cloudflare Enterprise account — required for proxy_read_timeout and Logpull entitlement
  • Enterprise plan must be pre-assigned by Cloudflare to your account — it cannot be set via API

System requirements

  • Python 3.9 or later
  • requests library: pip install requests

Files required

  • credentials.txt — in the same directory as the script (see Section 4)

After the script runs (manual steps)

  1. Log in to the Cloudflare dashboard
  2. Navigate to your zone → DNS → Nameservers
  3. Note the assigned Cloudflare nameservers (e.g. aria.ns.cloudflare.com)
  4. Update your domain registrar to use those nameservers
  5. Wait for DNS propagation (can take up to 24 hours)

3. Cloudflare API Token Permissions

Create a Custom Token with exactly these permissions. Using a Global API Key is not recommended for production.

CategoryPermissionAccess LevelRequired For
ZoneZoneEditCreating zones (Step 1)
ZoneDNSEditCreating A / CNAME records (Step 2)
ZoneZone SettingsEditTLS, HTTP/2, origin timeout, log retention (Steps 3–6)
ZoneFirewall ServicesEditDeploying WAF managed ruleset (Step 7)
ZoneCache RulesEditCreating cache rules via Rulesets API (Step 8)

How to create the token

  1. Go to Cloudflare Dashboard → My Profile → API Tokens
  2. Click Create Token → Custom Token
  3. Add all permissions from the table above
  4. Under Zone Resources: select All zones or specific zones in your account
  5. Under Account Resources: select your account for Account Settings Read
  6. Click Continue to Summary → Create Token
  7. Copy the token immediately — it is shown only once
Security: never commit credentials.txt to source control. Add credentials.txt to your .gitignore. The credentials.txt.example file is safe to commit.

4. credentials.txt Format

Create a file named credentials.txt in the same directory as the script. One key per line, KEY=VALUE with no spaces around =.

CLOUDFLARE_API_TOKEN=your-cloudflare-bearer-token
CLOUDFLARE_ACCOUNT_ID=your-account-id
SPLUNK_URL=https://your-splunk-instance:8088/services/collector/raw
SPLUNK_TOKEN=Splunk your-hec-token-here
DYNATRACE_URL=https://your-environment-id.live.dynatrace.com/api/v2/logs/ingest?header_Authorization=Api-Token%20your-dynatrace-token-here
KeyDescriptionExample / Notes
CLOUDFLARE_API_TOKEN Cloudflare API Bearer token Generated from My Profile → API Tokens. No prefix needed.
CLOUDFLARE_ACCOUNT_ID Your Cloudflare account ID Found in the dashboard URL: /accounts/{account_id}
SPLUNK_URL Full HEC endpoint URL including scheme Must use /services/collector/raw path. Port must be :443 or :8088.
SPLUNK_TOKEN Splunk HEC token with Splunk prefix Must be Splunk abc123... — the word "Splunk" followed by a space and the token.
DYNATRACE_URL Full Dynatrace log ingest URL with the API token embedded as a query parameter https://{env-id}.live.dynatrace.com/api/v2/logs/ingest?header_Authorization=Api-Token%20dt0c01.xxx — token requires logs.ingest scope. The space between Api-Token and the token value must be percent-encoded as %20.
Splunk channel IDs are auto-generated. The script generates a unique UUID for each of the three Splunk Logpush jobs at runtime using Python's uuid.uuid4(). You do not need to pre-create these.

5. Script Usage

Install dependency

pip install requests

Run the script

# A record (IP origin)
python cloudflare_zone_setup.py \
  --zone example.com \
  --record-type A \
  --origin 1.2.3.4

# CNAME record (hostname origin)
python cloudflare_zone_setup.py \
  --zone example.com \
  --record-type CNAME \
  --origin origin.example.com

6. Step-by-Step Walkthrough

1
Create Zone
POSThttps://api.cloudflare.com/client/v4/zones

Creates the zone under your account with type: full, meaning Cloudflare is the authoritative DNS. The Enterprise plan is inherited from the account — it is not specified in the API payload. jump_start: false prevents Cloudflare from auto-scanning existing DNS records.

On success: saves result.id as ZONE_ID, used by all subsequent steps.

On failure: no rollback needed (zone was not created), exits immediately.

2
Create DNS Record
POSThttps://api.cloudflare.com/client/v4/zones/{zone_id}/dns_records

Creates a proxied A or CNAME record for the root domain. proxied: true is mandatory — WAF, Logpush, and all Cloudflare security features only apply to traffic passing through the Cloudflare proxy. TTL is set to 1 (Auto) which Cloudflare enforces for all proxied records.

3
Set Minimum TLS Version to 1.2
PATCHhttps://api.cloudflare.com/client/v4/zones/{zone_id}/settings/min_tls_version

Rejects any HTTPS connections using TLS 1.0 or 1.1. Valid values: "1.0", "1.1", "1.2", "1.3". This is a zone-level setting that applies to all hostnames in the zone.

4
Disable HTTP/2 (Client → Edge)
PATCHhttps://api.cloudflare.com/client/v4/zones/{zone_id}/settings/http2

Disables HTTP/2 between clients and the Cloudflare edge. The documented API value is the string "off" — not the boolean false. This does not affect HTTP/2 from Cloudflare to the origin server.

5
Set Origin Read Timeout to 120 Seconds
PATCHhttps://api.cloudflare.com/client/v4/zones/{zone_id}/settings/proxy_read_timeout

Enterprise-only setting. Controls the maximum time Cloudflare waits for the origin to send a response after establishing a TCP connection. Default is 100 seconds. Valid range: 1–6000 seconds. Set to 120 as an integer (not a string).

6
Enable Log Retention
PUThttps://api.cloudflare.com/client/v4/zones/{zone_id}/logs/received/settings

Requires the Logpull entitlement (Enterprise). Retains HTTP request logs for up to 72 hours, queryable via the Logpull API (GET /zones/{id}/logs/received). This is separate from Logpush — Logpull lets you pull historical logs on demand.

7
Deploy WAF Managed Ruleset in Log Mode
PUThttps://api.cloudflare.com/client/v4/zones/{zone_id}/rulesets/phases/http_request_firewall_managed/entrypoint

Deploys the Cloudflare Managed Ruleset (efb7b8c949ac4650a09736fc376e9aee) in log-only mode. The overrides.rulesets[].action: "log" structure overrides every rule in the ruleset to log instead of block. Available actions: block, log, managed_challenge, challenge.

OWASP ruleset is not deployed. The OWASP Core Ruleset must be explicitly added as a separate rule in the same phase entrypoint if needed. This script only deploys the Cloudflare Managed Ruleset.
8
Cache Rule — Disable All Caching
PUThttps://api.cloudflare.com/client/v4/zones/{zone_id}/rulesets/phases/http_request_cache_settings/entrypoint

Uses the http_request_cache_settings phase to create a cache rule. "cache": false disables caching for all matched requests. "expression": "true" is the Rules Language expression that matches every request.

9
Logpush Job: HTTP Requests → Splunk
POSThttps://api.cloudflare.com/client/v4/zones/{zone_id}/logpush/jobs

Creates a Logpush job streaming HTTP request logs to Splunk HEC. Uses output_options.field_names array (current API approach, not the deprecated logpull_options string). A unique UUID channel ID is generated at runtime. The destination_conf uses the splunk:// scheme with the HEC endpoint stripped of its https:// prefix.

Endpoint must be /services/collector/raw. Supported ports: :443 or :8088.

10
Logpush Job: Firewall Events → Splunk
POSThttps://api.cloudflare.com/client/v4/zones/{zone_id}/logpush/jobs

Streams firewall event logs (WAF actions, rate limiting, etc.) to Splunk using dataset: "firewall_events". Uses a separate unique UUID channel ID. Includes FraudUserID field (added May 2026).

11
Logpush Job: DNS Logs → Splunk
POSThttps://api.cloudflare.com/client/v4/zones/{zone_id}/logpush/jobs

Streams DNS query logs to Splunk using dataset: "dns_logs". DNS logs are only available if the zone has active DNS traffic flowing through Cloudflare (requires proxied records and nameserver delegation).

12
Logpush Job: HTTP Requests → Dynatrace
POSThttps://api.cloudflare.com/client/v4/zones/{zone_id}/logpush/jobs

Streams a selected subset of HTTP request fields to Dynatrace Log Monitoring. Dynatrace requires output_type: "ndjson" with batch_prefix: "[", batch_suffix: "]", and record_delimiter: "," so that log batches are delivered as JSON arrays. The token must be prefixed with Api-Token and must have the logs.ingest scope.

The field list in the script is clearly marked with a comment for customisation. Full field reference: Cloudflare HTTP Requests log fields.

7. Rollback Behaviour

If any step fails (Cloudflare API returns "success": false), the script:

  1. Logs the full API error response to the terminal
  2. Sets the overall status to FAILED
  3. Calls DELETE /zones/{zone_id} to remove the partially-configured zone
  4. Generates the HTML report (including rollback status)
  5. Exits with code 1
Rollback deletes the zone entirely. All DNS records, settings, WAF rules, and Logpush jobs created before the failure point are removed along with the zone. This is intentional — a partial configuration is worse than no configuration.

What rollback does NOT cover

  • Step 1 failure — if zone creation itself fails, there is nothing to roll back
  • Registrar-level nameserver changes you may have already made manually

8. HTML Report

The script generates zone_report_{zone}_{timestamp}.html in the script directory on both success and failure.

Report sections

  • Header — zone name, zone ID, record type, origin, timestamp, and overall status badge (green for SUCCESS, red for FAILED)
  • Step Summary table — one row per step showing status (✓ / ✗ / —), API endpoint, and resource ID
  • Step Details — collapsible <details> element for each step showing the full raw API response JSON
  • Error Details (on failure) — highlighted section with the API error response that caused the failure
  • Rollback section (if triggered) — shows rollback status and the DELETE API response

9. Known Limitations

LimitationDetails
Maximum 4 Logpush jobs per zone This script creates exactly 4 (the maximum). Adding more Logpush jobs without deleting existing ones will fail. The script warns you before creating jobs.
NS delegation is manual Zone creation via API does not activate the zone. Nameserver updates at the registrar are required and cannot be automated via Cloudflare API.
Enterprise plan required proxy_read_timeout and Logpull are Enterprise-only features. The Enterprise plan must be pre-assigned by Cloudflare and cannot be set via API.
Logpull entitlement required Step 6 (log retention) will fail if the account does not have the Logpull entitlement. Contact your Cloudflare account team to enable it.
DNS logs require active traffic The Logpush DNS logs job (Step 11) only produces data once the zone is active (nameservers delegated) and receiving DNS queries.
OWASP ruleset not included The OWASP Core Ruleset is not deployed. It must be added separately with its own execute rule in the WAF phase entrypoint.
Single root record only The script creates one DNS record for the root domain. Additional records (www, subdomains) must be added separately.

10. Troubleshooting

Error / SymptomCauseFix
1001 — Invalid API token Token is malformed, expired, or lacks required permissions Recreate the token with all permissions in Section 3
1061 — Zone already exists The domain is already registered in Cloudflare (in this or another account) Delete or move the existing zone before running the script
7003 — Invalid value for field "type" Only "full" and "partial" are valid zone types Ensure the body is exactly as specified — no modifications
Step 5 fails — proxy_read_timeout not available Zone is not on an Enterprise plan Contact your Cloudflare account team to verify Enterprise entitlements
Step 6 fails — log retention not available Logpull entitlement not enabled on the account Contact Cloudflare support to enable the Logpull entitlement
Step 9–12 fail — 1009 max Logpush jobs reached Zone already has 4 Logpush jobs from a previous run Delete existing jobs via DELETE /zones/{id}/logpush/jobs/{job_id} then re-run
Splunk Logpush job created but no data arrives HEC token inactive, wrong endpoint, or zone not yet active Verify HEC token is enabled in Splunk, endpoint is /services/collector/raw, and nameservers are delegated
Dynatrace Logpush job created but data is not parsed as JSON Missing output_type: ndjson + batch options Ensure the job body includes output_type, batch_prefix, batch_suffix, and record_delimiter exactly as specified
requests.exceptions.ConnectionError No network access to api.cloudflare.com Check firewall/proxy settings; ensure outbound HTTPS to api.cloudflare.com:443 is allowed
Missing credential error on startup credentials.txt is missing a required key Check against Section 4 — all six keys must be present with non-empty values
